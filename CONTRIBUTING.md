# Contribución

## Commits

Usar mensajes simples y descriptivos, por ejemplo:

- `docs: actualizar documentación de arquitectura`
- `feat: preparar endpoint de salud`
- `test: agregar prueba de integración de health`

## Reglas de trabajo

- No subir secretos, credenciales, tokens, certificados ni cadenas de conexión reales.
- Revisar `git status --short` antes de comenzar a trabajar.
- Mantener cambios limitados al prompt o tarea asignada.
- En el futuro, usar ramas `feature/*` para incorporar colaboradores.
- No hacer push directo a `main`; `main` será la rama estable.
- Abrir Pull Request antes de merge.
- Respetar los mockups oficiales en futuros ajustes de frontend.
